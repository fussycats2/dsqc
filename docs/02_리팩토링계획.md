# 리팩토링 계획서 — 엑셀 매크로 → 웹앱

> 스택: **Next.js 15 (App Router) + Supabase + Vercel** · 목표 비용 **$0 (무료 티어)**
>
> **확정 사항 (2026-05-29)**
> - 진행: **계획 검토/수정 우선** (스캐폴딩은 계획 확정 후)
> - 권한: **단일 사업장 공용 계정 1개** (부서/역할 분리 없음 → RLS 최소화)
> - 인쇄: **PDF/브라우저 화면 인쇄로 충분** (종이 양식 픽셀 재현 불필요)
> - 데이터 모델: 입고/출고는 1:1 전이 아님 → **lots + lot_links(계보)** 구조로 병합/분할 표현
> - **공정 스키마 2계열**: `io`(일반 공정 + **검수**) / `work`(연마·뻥·빠우). 검수는 일반과 같은 입고/출고형.
>   저장은 공통 `lots`, 사용 컬럼·UI·로스식만 `schema_type`로 분기 (io=Tag로스, work=작업전−작업후)
> - 이전파트 추적: **계보 히스토리 화면(`/lots/[id]`)** 으로 제공
> - 집계 표기: 괄호 문자열 일련번호 미재현 → **관계 + UI**로 단순화
> - 마감 = **일마감 + 월마감** 2종. 목적은 **미작업(재고) 이월**. 일마감=미작업분을 다음날로 이월하며
>   완료분 정리·일일 스냅샷, 월마감=한 달치 파일 내보내기 + 서버 정리(직전 1개월 보존)
> - 마감 파일 양식: **무관**(기록·백업용)
> - 동시 사용: PC **3대**, 동시 편집은 드물지만 가끔 발생 → 무거운 잠금 대신 **낙관적 동시성(행 버전) + 실시간 갱신**
> - 품질결산서: **기존 로직 유지** — LAN 공유의 `품질결산서.xlsm`을 직접 조작해야 함 → **로컬 브릿지** 필요(아래 7-1)
> - 사용 환경: **데스크톱(24~27인치) 전용**, 모바일/태블릿 없음 → 반응형 비우선, 고밀도 그리드 최적화

## 1. 목표와 원칙

- 엑셀 .xlsm의 모든 운영 기능을 웹앱으로 이전 (입고→투입→이관→출고→검수→집계→인쇄→마감)
- **시트=테이블 복제 구조를 1개의 정규화 DB로 통합** → 공정 추가는 데이터 한 줄로
- 동시 다중 사용자, 실시간 중량오류 경고, 원격/모바일 접근
- 비즈니스 규칙(8장)을 서버측 단일 소스로 고정

## 3. 데이터 모델 (Postgres)

핵심: 44개 시트를 **공정 마스터 + 작업항목 + 이동로그**로 정규화.

```sql
-- 공정 마스터 (44시트 → 행)
processes (
  id            uuid pk,
  name          text,            -- '기계','빠우(할로우)','검수(기계)'…
  code          text,            -- 'M','C','QM' (일련번호 약어)
  karat         text,            -- '18K' | '14K'
  category      text[],          -- 메뉴 카테고리(태그)
  schema_type   text,            -- ★ 'io'(입고/출고형: 일반+검수) | 'work'(작업중/완료형: 연마/뻥/빠우) | 'entry'(작성)
  is_inspection boolean,         -- 검수 여부(io형이지만 흐름상 최종단계)
  is_blue       boolean,         -- 14K 파란색 표기
  sort_order    int
)
-- schema_type별 차이:
--   io   : 출고측에 실중량/Tag수정/Tag중량/Tag로스/출고중량. 로스=Tag로스(X), 출고중량=실중량+Tag-Tag중량
--   work : 출고측에 작업전/작업후/로스/로스율. 로스=작업전-작업후, Tag보정 없음
--   → lots 테이블은 공통(아래), 사용 컬럼·UI·검증식만 schema_type로 분기

-- 작업 단위 (공정 시트의 "한 행" = 입고측 또는 출고측 한 건)
--  ※ 입고와 출고는 1:1 상태전이가 아님. 집계(병합) = 입고 N건 → 출고 1건,
--    분할 = 입고 1건 → 출고 N건. 그래서 입고/출고를 별도 행(lot)으로 두고
--    관계는 lot_links(계보)로 표현한다.
lots (
  id            uuid pk,
  serial        text,            -- 'M_260521_001' (병합/분할 시 새 일련번호 부여)
  process_id    uuid → processes,
  side          text,            -- 'in'(입고/작업중) | 'out'(출고/완료)
  description   text,            -- 내역
  qty           numeric,         -- 수량
  weight        numeric,         -- 입고측=입중량 / 출고측= io:실중량 · work:작업후
  weight_before numeric,         -- work형 전용: 작업전 (로스=weight_before-weight)
  tag           numeric,         -- Tag (1tag=0.035g)
  tag_fixed     numeric,         -- io형: Tag수정 (출고)
  tag_weight    numeric,         -- io형: Tag중량 (출고)
  tag_loss      numeric,         -- io형: Tag로스 (출고)
  q             numeric,
  due_date      date,            -- 납기
  raw_weight    numeric,         -- 원중량
  note          text,            -- 비고
  status        text,            -- '대기'|'작업중'|'완료'
  period_id     uuid → periods,  -- 마감 구분(=현재 월)
  created_at    timestamptz,
  completed_at  timestamptz,
  updated_at    timestamptz,     -- 낙관적 동시성(3대 동시편집 충돌 감지)
  version       int              -- 저장 시 버전 비교
)
-- 생성열/뷰 (schema_type별로 다름):
--   io형  : 출고중량 = weight + tag - tag_weight (엑셀 Y=O+P-W), 로스 = tag_loss(X)
--   work형: 로스 = 작업전(weight_before) - 작업후(weight=작업후), 로스율 = 1 - 작업후/작업전
--           ※ work형은 weight_before(작업전) 컬럼을 추가로 사용, tag_weight/tag_fixed 미사용

-- 계보(lineage): 점1·점2를 모두 해결하는 핵심 테이블
--   move : 출고 lot → 다음 공정 입고 lot (이전파트 = from의 process)
--   merge: 입고 N lot → 출고 1 lot (집계)
--   split: 입고 1 lot → 출고 N lot (수량 분할)
lot_links (
  id        uuid pk,
  from_lot  uuid → lots,   -- 원천(입고/이전공정 출고)
  to_lot    uuid → lots,   -- 결과(출고/현 공정 입고)
  relation  text           -- 'move' | 'merge' | 'split'
)
--  → 어떤 lot의 from_lot 체인을 거슬러 올라가면 "이전파트 히스토리"가 그래프로 나온다.
--    UI는 이 체인을 타임라인/계보 트리로 렌더.

-- 이동/입출고 로그 (입출로그 시트, 사람이 읽는 감사 기록)
movements (
  id            uuid pk,
  ts            timestamptz,
  type          text,            -- '입고'|'출고'|'투입'|'이관'|'타부서투입'|'집계'|'분할'
  source_process_id uuid,
  target_process_id uuid,
  lot_id        uuid → lots,
  qty numeric, weight numeric, tag numeric, q numeric,
  due_date date, raw_weight numeric, note text,
  actor_name    text             -- 공용계정이므로 작업자명 텍스트만(선택)
)

-- 월마감 구분 (스냅샷은 서버에 보관하지 않음 → 점4 참고)
periods ( id uuid pk, label text, opened_at, closed_at, status ) -- 'open'|'closed'

-- 공휴일(날짜선택기), 앱설정
holidays ( date date pk, name text )
settings ( key text pk, value jsonb )   -- TAG_PER_GRAM=0.035 등
```

집계 시트(raw/sum)는 **SQL 뷰**로 대체:
- `v_process_balance` (파트별 입고/출고 합계)
- `v_weight_reconcile` (중량오류 = 입고 − 출고 − 로스 ≠ 0)
- `v_inspection_status` (검수 현황)

서버 함수(Postgres RPC / Server Action)로 이식할 핵심 로직:
- `next_serial(process_id)` — 약어_YYMMDD_일별순번 (트랜잭션 내 max+1)
- `move_lots(lot_ids[], target_process, type)` — 투입/이관/출고 (다음 공정 입고 lot 생성 + `lot_links(move)` + 로그)
- `merge_lots(lot_ids[])` — 집계 (입고 N → 출고 1, `lot_links(merge)`)
- `split_lot(lot_id, n)` — 수량 분할 (`lot_links(split)`)
- `close_period()` — 마감: 현재 기간 데이터를 **파일로 내보낸 뒤 서버에서 삭제**(점4)

## 4. 화면 설계 (라우트)

```

/entry                 작성 (주문 입력 + Tag검증 + 보내기/입고)
/process/[id]          공정 시트 (입고·출고 그리드, 행선택 → 투입/이관/출고/집계/분할/삭제)
/lots/[id]             ★ 항목 계보(히스토리) — 이전파트부터 현재까지 흐름·중량·로스 추적 (점2)
/log                   입출로그 (필터/검색)

/reports/quality       품질결산서

```

- 그리드: 셀 편집 + 다중행 선택 → 우측/상단 액션바(보내기·투입·이관·출고·집계·분할·삭제·완료).
- **계보 히스토리(점2)**: 일련번호/행 클릭 → `lot_links`를 거슬러 "이전파트 → … → 현재" 흐름을
  타임라인/트리로 표시. 각 단계 중량·로스·작업시간 포함. (엑셀의 U열 "이전파트"를 살아있는 추적으로 대체)
- **집계(병합) 단순화(점3)**: 엑셀의 `T_260520_(003,004)` 식 괄호 문자열 표기는 **재현하지 않음**.
  병합 출고 lot은 깔끔한 새 일련번호를 받고, "어떤 입고를 묶었는지"는 `lot_links(merge)` 관계로 보관 →
  UI에서 "입고 3건 병합 (003,004,005)"처럼 자연스럽게 펼쳐 보여줌. 일련번호 자체는 단순하게 유지.
- 로스 HUD → 모달 계산기 컴포넌트, Tag 보정 → 인라인 액션.
- 날짜선택기 → 공휴일 표시 DatePicker 컴포넌트.
- 중량오류 경고 → Supabase Realtime 구독으로 실시간 배너.
- 편집모드 → 공용계정이므로 권한이 아닌 **UI 토글**(실수 방지용 읽기/편집 잠금)로 단순화.
- 인쇄(확정) → 장부 라우트를 **인쇄용 CSS(@media print) + 브라우저 인쇄/PDF 저장**으로 처리.
  종이 양식 픽셀 재현은 하지 않음. 필요 시 `@react-pdf/renderer`로 PDF 다운로드 버튼만 추가.

### 마감 동작 — 일마감 + 월마감 (확정)

마감의 진짜 목적: **미작업으로 남은 항목(재고)을 다음 기간으로 이월**하고, 끝난 것(완료·출고분)은
기록으로 빼내 화면을 깨끗이 비우기. 핵심은 lots 모델에서 **미작업 lot을 지우지 않는 것** = 자동 이월.

**일마감 (매일)**
```
완료·출고된 lot → 일일 기록으로 보관 + 현재 화면에서 정리
미작업(재고) lot → 다음날로 이월(서버 유지 = 다음날 시작 재고)
일일 집계 스냅샷(파트별 입출고/로스) 저장
```

**월마감 (매월)**
```
[관리자] /admin/period → "월마감"
   → 한 달치 데이터를 파일로 내보내 다운로드 (양식 무관: 기록/백업용)
   → 다운로드 성공 확인 후: 해당 월 데이터 서버에서 정리(직전 1개월은 안전망으로 보존)
   → 미작업 재고는 계속 이월, 새 월 period 시작
```
- 서버 보존 = **현재 운영분 + 직전 1개월** → DB 항상 수MB, 무료 한도 무관.
- 실수 방지: 내보내기 다운로드 성공 확인 전에는 정리 안 함(2단계 확인).

### 동시 사용 (확정)

PC 3대, 동시 편집은 드물지만 가끔 발생 → **낙관적 동시성**으로 충분:
- `lots`에 `updated_at`/`version` 두고, 저장 시 버전 불일치면 "다른 사람이 먼저 수정함" 안내 후 새로고침.
- Supabase **Realtime 구독**으로 같은 공정 그리드를 보는 다른 PC에 변경 즉시 반영(충돌 빈도 자체를 낮춤).
- 무거운 행 잠금/비관적 락은 불필요.

### 7-1. 품질결산서 전송 — 외부 Excel 직접 조작(기존 로직 유지)

기존 매크로(Module37)는 LAN 공유 `\\192.168.0.70\...\품질결산서.xlsm`의 `일일결산서` 시트
특정 셀에 raw 집계값을 매핑해 써넣는다. **브라우저는 이 네트워크 경로에 접근 불가** → 두 방법:

1. **(권장) 로컬 브릿지 스크립트**: 그 LAN에 있는 PC에서 도는 작은 스크립트(Python+openpyxl 또는
   PowerShell)가 Supabase에서 raw 집계를 읽어와 **기존과 동일한 셀 매핑**으로 `품질결산서.xlsm`에 기록.
   = 기존 "조작" 로직을 그대로 보존하면서 데이터 출처만 엑셀→DB로 교체. 버튼 한 번/스케줄 실행.
2. (대안) 웹에서 매핑값을 담은 파일을 받아 그 PC에서 수동 반영 — 자동화 약함.

→ 매핑 좌표(O3:O7→C5:G5 등)는 Module37에 그대로 있으므로 1:1 이식 가능.

## 6. 데이터 마이그레이션

- 본 분석에서 이미 `openpyxl`로 전 시트 파싱 가능 확인 (현재 ~620 공정행, 52 로그행).
- 1회성 Python 스크립트: 시트별 13행~ 파싱 → `lots`/`movements`/`lot_links` insert.
  일련번호·이전파트(U열)·상태·색상마킹을 컬럼/관계로 변환 (이전파트는 가능한 범위에서 `lot_links(move)` 복원).
- 검증: 마이그레이션 후 `v_process_balance` 합계가 엑셀 raw 시트와 일치하는지 대조.
